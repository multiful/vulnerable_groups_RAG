4. 추천 매핑 예시

[IT/디지털]
- 데이터 분석가 되고 싶어 -> 세부: 데이터/AI
- 백엔드 개발 쪽 취업하고 싶어 -> 세부: 소프트웨어개발
- 보안 쪽으로 가고 싶어 -> 세부: IT인프라/보안
- 측량·GIS 쪽 취업하고 싶어 -> 세부: 토목/측량/공간정보

[엔지니어링/산업기술]
- 전기기사 따서 취업하고 싶어 -> 세부: 전기/전자
- 기계설계 쪽으로 가고 싶어 -> 세부: 기계/제조
- 건축 쪽 취업하고 싶어 -> 세부: 건축/실내건축
- 안전관리자 하고 싶어 -> 세부: 환경/안전
- 주조·금속 쪽 자격증 추천해줘 -> 세부: 재료/금속
- 화학분석 쪽 가고 싶어 -> 세부: 화학/바이오
- 원자력·방사선 쪽 취업 -> 세부: 에너지/원자력

[경영/비즈니스]
- 회계팀 취업하려면 뭐 따야 해? -> 세부: 금융/회계
- 사무직 취업 준비하고 싶어 -> 세부: 경영/사무
- 무역회사 들어가고 싶어 -> 세부: 유통/물류/무역
- 공기업 사무직 준비 중이야 -> 세부: 공공/행정
- CS·고객응대 쪽 취업 -> 세부: 영업/CS

8. 도메인 2개 허용 여부 (검토)

현재 구조:
- certificates_vectors.domain: 단일 TEXT (세부 도메인 1개)
- 쿼리 슬롯 "도메인": 이미 복수 지원 (쉼표 구분 문자열, metadata_soft_score에서 토큰 overlap으로 가산)

도메인 2개 허용 시:
- 장점: 일부 자격증이 복합 도메인(예: 정보처리+경영, 건설+환경안전)일 때 검색·추천 recall 향상 가능.
- 단점: 스키마 변경(domain을 TEXT[] 또는 comma-separated로 저장), 매핑 규칙·overrides 설계 복잡도 증가, 메타데이터 소프트 스코어는 이미 overlap 기반이라 다중 값이면 이득 있으나 우선순위/가중치 정책 필요.

의견: 단기에는 세부 라벨 1개 유지가 운영·평가하기 쉽다. 복수 도메인이 꼭 필요해지면, (1) 우선 "주 도메인" 1개만 저장하고 (2) metadata에 secondary_domain 또는 domains[]를 옵션으로 두어, 소프트 스코어에서만 overlap 시 가산하는 방식으로 확장하는 것을 권장한다.
