## 1. 문서 목적

이 문서는 프로젝트 루트 문서를 **어떻게 유지, 수정, 정렬할지**에 대한 운영 기준을 정의한다.  
본 프로젝트는 코드보다 문서를 우선하며, 루트 문서는 프로젝트의 단일 기준선(single source of truth) 역할을 한다.

이 문서의 목적은 다음과 같다.

- 문서 갱신 순서를 고정한다.
- 구현 전 어떤 문서를 먼저 확인해야 하는지 정의한다.
- 해결된 문제와 새 문제를 PRD에 어떻게 반영할지 정한다.
- 문서 간 충돌이 발생했을 때 정리 기준을 제공한다.
- 문서를 수정할 때 상단 메타데이터도 함께 최신 상태로 유지하도록 강제한다.
- 문서 우선순위 숫자와 실제 우선순위 목록이 항상 일치하도록 강제한다.
- 모든 md 문서에 해시 라인을 강제하고, 변경 감지와 증분 처리 기준을 일관되게 유지하도록 한다.

이 문서는 **운영 규칙 문서**이며, 시스템 구조 자체는 `SYSTEM_ARCHITECTURE.md`, 디렉토리 구조 자체는 `DIRECTORY_SPEC.md`, 제품 요구사항 자체는 `PRD.md`에서 담당한다.

## 2. 핵심 원칙

1. 구현 전에 관련 루트 문서를 먼저 읽는다.
2. 기능, 정책, 구조가 바뀌면 코드보다 먼저 문서를 수정한다.
3. 해결된 문제는 PRD에 그대로 방치하지 않는다.
4. 루트 문서 간 불일치가 생기면 즉시 정리한다.
5. 문서 수정 없이 구현만 먼저 진행하는 것을 금지한다.
6. 루트 문서를 수정할 때는 **본문만이 아니라 문서 상단 메타데이터도 함께 갱신**한다.
7. 문서 메타데이터의 `문서 우선순위` 값은 **유일해야 하며**, 본 문서의 우선순위 목록과 일치해야 한다.
8. 모든 루트 md 문서는 **문서 해시 라인**을 반드시 포함해야 한다.
9. 해시 라인은 단순 표기가 아니라 변경 감지, 선택적 재처리, 캐시 재사용 기준으로 활용한다.

## 4. 문서 상단 메타데이터 관리 규칙

모든 루트 문서는 문서 맨 위에 아래 메타데이터를 유지한다.

- 파일명
- 최종 수정일
- 문서 해시
- 문서 역할
- 문서 우선순위
- 연관 문서
- 참조 규칙

### 4.1 문서 해시 라인 형식
모든 md 파일은 아래 형식의 해시 라인을 사용한다.

- `> **문서 해시**: SHA256:<AUTO_HASH_OR_TBD>`

문서 해시 라인은 `최종 수정일` 바로 아래에 둔다.

### 4.2 수정 시 필수 갱신 항목
문서를 수정하면 아래 항목을 반드시 함께 점검한다.

1. `최종 수정일`을 실제 수정일로 갱신한다.
2. 문서 해시를 실제 계산값 또는 허용 placeholder 상태로 갱신한다.
3. 문서 역할이 달라졌다면 `문서 역할`을 갱신한다.
4. 참조 대상이 달라졌다면 `연관 문서`를 갱신한다.
5. 문서 우선순위가 변하면 `문서 우선순위`를 갱신한다.
6. 참조 절차가 바뀌면 `참조 규칙`을 갱신한다.

### 4.3 해시 계산 원칙
- 해시는 해시 라인 자기 자신을 포함해서 계산하지 않는다.
- 필요하면 `최종 수정일` 라인도 해시 계산에서 제외할 수 있다.
- 목적은 의미 내용이 바뀌었을 때만 해시가 바뀌게 하는 것이다.
- 실제 해시를 계산할 수 없으면 `SHA256:TBD` 또는 `SHA256:<AUTO_HASH_OR_TBD>`를 사용하되, 가짜 계산값을 임의 기입하지 않는다.

### 4.4 원칙
- 본문만 수정하고 메타데이터를 방치하지 않는다.
- `최종 수정일`은 루트 문서의 최신성을 판단하는 기본 기준으로 본다.
- `문서 해시`는 변경 감지와 선택적 재처리의 기준으로 사용한다.
- 메타데이터가 오래되면 문서도 최신 기준 문서로 간주하지 않는다.

## 6. PRD 갱신 원칙

PRD는 **현재 제품이 해결해야 하는 문제와 범위**만 남겨야 한다.

### 6.1 해결된 문제 처리
해결이 끝난 문제는 아래 중 하나를 반드시 수행한다.

#### 방식 A. Active / Resolved 분리
- 진행 중 문제 → `Active Problems`
- 해결 완료 문제 → `Resolved Problems`

#### 방식 B. PRD 본문에서 제거 + 로그 기록
- 더 이상 핵심 제품 문제가 아니면 PRD 본문에서 제거
- 대신 `DEV_LOG.md` 또는 별도 변경 이력에 기록

### 6.2 새 문제 추가
새로운 핵심 문제가 발견되면 먼저 PRD를 갱신한 뒤 시스템/기능/데이터 문서를 수정한다.

### 6.3 원칙
- PRD에는 현재 중요한 문제만 남긴다.
- 이미 해결된 문제를 핵심 문제로 계속 유지하지 않는다.
- 구현이 PRD보다 먼저 앞서가지 않게 한다.

## 10. Cursor / AI 작업 원칙

AI 또는 Cursor를 활용한 작업은 아래 원칙을 따른다.

1. 작업 전 `README.md`, `CHANGE_CONTROL.md`, 관련 루트 문서를 먼저 읽는다.
2. 루트 문서와 충돌하는 구현 제안은 보류한다.
3. 루트 문서에 없는 새 개념을 임의로 코드에 먼저 넣지 않는다.
4. 문서 리팩토링이 필요한 경우, 구현보다 먼저 문서 수정안을 제안한다.
5. 해결 완료된 문제는 PRD 기준으로 상태를 갱신한다.
6. 디렉토리 구조 변경 시 `DIRECTORY_SPEC.md`를 먼저 수정한다.
7. 루트 문서를 수정할 경우 상단 메타데이터의 `최종 수정일`도 함께 갱신한다.
8. 우선순위 숫자가 중복되면 메타데이터를 즉시 정정한다.
9. 새 md 문서를 만들 때는 문서 해시 라인을 반드시 넣는다.
10. 실제 해시 미계산 시 placeholder/TBD 규칙을 따른다.
11. unchanged 입력에 대해 전체 재처리보다 증분 처리 방식을 우선 제안한다.

## 11. 문서 리팩토링 원칙

1. 문서는 서로 역할이 겹치지 않게 유지한다.
2. README는 요약/입구 역할만 가진다.
3. PRD는 제품 문제와 범위에 집중한다.
4. SYSTEM_ARCHITECTURE는 구조와 계층에 집중한다.
5. FEATURE_SPEC은 기능 입출력과 예외처리에 집중한다.
6. DATA_SCHEMA는 엔티티/관계/필드에 집중한다.
7. API_SPEC은 계약과 오류 형식에 집중한다.
8. PROMPT_DESIGN은 프롬프트 입력/출력과 가드레일에 집중한다.
9. RAG_PIPELINE은 parse~retrieval 흐름에 집중한다.
10. DIRECTORY_SPEC은 폴더와 파일 구조에만 집중한다.
11. CHANGE_CONTROL은 갱신 절차와 운영 규칙에만 집중한다.

즉, 문서가 커질수록 **역할별 분리**를 더 강하게 유지해야 한다.

## 12. 해시 라인과 증분 처리 운영 원칙

- 해시 라인은 단순 형식 통일이 아니라 변경 감지 기준이다.
- 해시 라인은 선택적 재처리와 캐시 재사용의 출발점이다.
- 전체 rebuild는 마지막 수단이며, 가능하면 변경된 단계만 재실행한다.
- parser/chunker/embedding 규칙이 바뀌면 해당 stage version도 함께 관리한다.
- 상세 증분 빌드 규칙은 `HASH_INCREMENTAL_BUILD_GUIDE.md`를 따른다.

## 13. 최종 원칙

이 프로젝트는 **문서를 먼저 고치고, 그 다음 구현한다**를 기본 원칙으로 한다.  
문서가 최신 상태가 아니면 구현도 최신 상태가 아닌 것으로 간주한다.

또한 문제 해결 이후에는 PRD를 반드시 갱신하고, 구조 변경 이후에는 관련 구조 문서를 반드시 갱신하며, 루트 문서를 수정할 때는 상단 메타데이터의 `최종 수정일`과 `문서 해시`도 반드시 함께 갱신한다.
