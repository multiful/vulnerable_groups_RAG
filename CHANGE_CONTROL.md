# CHANGE_CONTROL.md

## 1. 문서 목적

이 문서는 프로젝트의 루트 문서를 어떻게 유지·갱신할지에 대한 운영 기준을 정의한다.  
본 프로젝트는 코드보다 문서를 우선하며, 루트 문서는 프로젝트의 단일 기준선(single source of truth) 역할을 한다.

---

## 2. 핵심 원칙

1. 기능 구현 전에 관련 루트 문서를 먼저 확인한다.
2. 기능/정책/구조가 바뀌면 코드보다 먼저 루트 문서를 수정한다.
3. 해결된 문제는 PRD에서 그대로 방치하지 않는다.
4. PRD, 아키텍처, 기능 명세, 데이터 스키마 간 불일치가 생기면 즉시 정리한다.
5. 문서 수정 없이 구현만 먼저 진행하는 것을 금지한다.

---

## 3. 문서 역할 구분

### PRD.md
- 현재 해결해야 하는 문제
- 타깃 사용자
- 핵심 기능
- 제품 범위 / 비범위

### SYSTEM_ARCHITECTURE.md
- 시스템 구성
- 데이터 흐름
- 프론트/백엔드/AI 역할

### FEATURE_SPEC.md
- 기능별 입력 / 출력 / 예외처리

### DATA_SCHEMA.md
- 데이터 구조
- canonical schema
- 주요 필드 정의

### CHANGE_CONTROL.md
- 문서 갱신 원칙
- 문서 수정 순서
- 해결된 문제의 처리 방식
- 진행 전 확인 체크리스트

---

## 4. 해결된 문제 처리 원칙

PRD에 적힌 문제나 요구사항이 해결되었을 경우, 아래 중 하나를 반드시 수행한다.

### 방식 A. Active / Resolved 분리
- 해결 전 문제는 `Active Problems`
- 해결 완료 문제는 `Resolved Problems`로 이동

### 방식 B. 해결 완료 후 삭제 + 변경 이력 기록
- 현재 더 이상 제품 핵심 문제가 아니면 PRD 본문에서 제거
- 대신 DEV_LOG 또는 CHANGE_CONTROL의 변경 이력에 기록

기본 원칙은 다음과 같다.

- **현재 제품이 해결해야 하는 문제만 PRD에 남긴다.**
- 해결이 끝난 문제를 PRD의 핵심 문제 섹션에 계속 두지 않는다.
- 새롭게 중요한 문제가 생기면 PRD를 갱신한다.

---

## 6. 작업 시작 전 체크리스트

새 작업을 시작하기 전에 아래를 확인한다.

- [ ] README.md를 읽었는가
- [ ] CHANGE_CONTROL.md를 읽었는가
- [ ] 이번 작업과 관련된 PRD 요구사항이 최신인가
- [ ] 관련 아키텍처/기능/스키마 문서가 최신인가
- [ ] 이미 해결된 문제를 다시 구현하려는 것은 아닌가
- [ ] 새 문제 또는 새 범위가 생겼다면 먼저 문서에 반영했는가

---

## 7. 작업 완료 후 체크리스트

작업이 끝난 뒤 아래를 확인한다.

- [ ] 구현 내용이 루트 문서와 일치하는가
- [ ] 해결된 문제를 PRD에서 업데이트했는가
- [ ] 구조 변경이 있으면 SYSTEM_ARCHITECTURE.md를 수정했는가
- [ ] 기능 변경이 있으면 FEATURE_SPEC.md를 수정했는가
- [ ] 데이터 구조가 바뀌면 DATA_SCHEMA.md를 수정했는가
- [ ] 로그를 DEV_LOG.md에 남겼는가

---

## 8. Cursor / AI 작업 원칙

AI 또는 Cursor를 활용한 작업은 아래 원칙을 따른다.

1. 작업 전 `README.md`, `CHANGE_CONTROL.md`, 관련 루트 문서를 먼저 읽는다.
2. 루트 문서와 충돌하는 구현 제안은 보류한다.
3. 루트 문서에 없는 새 개념을 임의로 코드에 넣지 않는다.
4. 해결 완료된 문제는 PRD 기준으로 상태를 갱신한다.
5. 문서 리팩토링이 필요한 경우, 구현보다 먼저 문서 수정안을 제안한다.

---

## 9. 최종 원칙

이 프로젝트는 “문서를 먼저 고치고, 그 다음 구현한다”를 기본 원칙으로 한다.  
문서가 최신 상태가 아니면 구현도 최신 상태가 아닌 것으로 간주한다.
