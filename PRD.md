# PRD.md

## 1. 문서 목적

이 문서는 **청년 위험군 맞춤형 자격증·로드맵 추천 시스템**의 제품 요구사항(Product Requirements)을 정의한다.  
본 문서는 문제 정의, 타깃 사용자, 핵심 사용자 가치, 기능 범위, 사용자 시나리오, 성공 기준, 비범위를 명확히 하여 이후 시스템 아키텍처, 기능 명세, 데이터 스키마, API 설계의 상위 기준 문서로 사용한다.

이 문서의 목적은 다음과 같다.

- 우리가 해결하려는 문제를 명확히 정의한다.
- 누구를 위한 제품인지 구체화한다.
- 어떤 기능을 우선 구현하고, 무엇을 후속으로 미룰지 결정한다.
- 기술 설계가 아니라 **제품 목표와 제품 범위**를 고정한다.

---

## 2. 프로젝트 한 줄 정의

청년의 위험군 단계와 관심 직무/도메인을 바탕으로,  
**관련 자격증을 추천하고 → 단계에 맞는 로드맵을 제안하며 → 추후 시험 일정/접수 일정/지원 링크까지 연결하는 맞춤형 추천 시스템**을 구축한다.

---

## 3. 문제 정의

### 3.1 현재 문제
기존의 자격증 탐색 방식은 다음과 같은 문제를 가진다.

1. 사용자의 현재 상태를 반영하지 못한다.  
   - 단순히 “무슨 자격증이 유명한가” 수준의 추천에 머무른다.
   - 취업 준비 상태, 위험 수준, 준비 여건 차이를 반영하지 못한다.

2. 자격증과 직무, 도메인, 로드맵이 분리되어 있다.  
   - 자격증 정보는 있어도 어떤 직무와 연결되는지 불명확하다.
   - 어떤 단계의 사용자에게 적합한지 설명이 부족하다.

3. 공식 정보가 여러 소스에 흩어져 있다.  
   - PDF 안내문, HTML 페이지, CSV 매핑 데이터, 기관별 일정 정보가 분리되어 있다.
   - 사용자는 하나의 흐름 안에서 탐색하기 어렵다.

4. 단순 검색만으로는 “다음 행동”을 제시하기 어렵다.  
   - 사용자는 자격증 이름만 아는 것이 아니라,
     “내 상황에서 지금 뭘 해야 하는가”를 알고 싶어 한다.

## 4. 제품 목표

본 제품의 목표는 아래 3가지를 하나의 흐름으로 연결하는 것이다.

### 4.1 위험군 기반 개인화
사용자를 1단계~5단계 위험군 체계로 관리하고,  
단계에 따라 추천 우선순위와 로드맵 구조를 다르게 제공한다.

### 4.2 자격증 추천의 구조화
자격증을 단일 객체가 아니라 다음 관계 안에서 추천한다.

- 자격증 ↔ 직무
- 자격증 ↔ 도메인
- 위험군 ↔ 로드맵 단계
- 추후 자격증 ↔ 시험 일정 / 접수 일정 / 지원 링크

### 4.3 행동 가능한 다음 단계 제시
단순 추천 결과가 아니라,
- 왜 이 자격증이 적합한지
- 어떤 직무/도메인과 연결되는지
- 어떤 로드맵 순서로 준비해야 하는지
를 함께 제시한다.

---

## 5. 타깃 사용자

## 5.1 1차 타깃
취업 또는 진로 준비 중인 청년 사용자

예:
- 취업 준비를 시작했지만 어떤 방향으로 가야 할지 모르는 사용자
- 하고 싶은 직무는 있지만 어떤 자격증부터 준비해야 할지 모르는 사용자
- 취업 의지는 있으나 준비 수준이나 환경 제약으로 인해 바로 취업이 어려운 사용자

## 5.2 2차 타깃
자격증/직무/도메인 정보를 구조화해 관리하려는 운영자 또는 관리자

예:
- 자격증-직무-도메인 매핑 데이터를 정리하는 관리자
- 위험군 단계별 로드맵 기준을 운영하는 기획자
- 추천 규칙과 데이터 품질을 검토하는 협업자

## 6. 사용자 가치

이 시스템이 사용자에게 제공해야 하는 핵심 가치는 다음과 같다.

### 6.1 내 상태에 맞는 추천
사용자는 “누구에게나 좋은 자격증”이 아니라,  
**현재 내 단계에서 의미 있는 자격증**을 추천받아야 한다.

### 6.2 이해 가능한 연결
사용자는 추천 결과를 볼 때,
- 어떤 직무와 연결되는지
- 어떤 도메인에 속하는지
- 왜 지금 추천되는지
를 이해할 수 있어야 한다.

### 6.3 다음 행동의 명확성
사용자는 추천 결과를 본 뒤
- 당장 준비할 것
- 다음에 준비할 것
- 추후 확인할 일정/링크
를 순서대로 이해할 수 있어야 한다.

## 7. 핵심 제품 개념

## 7.1 위험군 단계 체계
위험군은 **1단계 ~ 5단계** 구조로 관리한다.

- **1단계**: 취업 안정권
- **5단계**: 취업을 하고 싶어도 하기 어려운 가장 높은 위험군
- **2~4단계**: 후속 문서에서 세부 의미 확정 예정

현재 PRD에서는 위험군을 “단계형 개인화 기준”으로 정의하며,
2~4단계의 구체 규칙은 데이터/정책 문서에서 확정한다.

---

## 7.2 도메인 / 직무 taxonomy 고정
추천 결과의 정합성을 위해 자유 텍스트 라벨을 허용하지 않는다.

- `related_domains`는 **domain taxonomy의 세부 라벨만 사용**
- `related_jobs`는 **희망 직무 taxonomy의 세부 직무만 사용**

즉, 추천 결과는 임의의 새 라벨을 생성하지 않고, 사전에 정의된 taxonomy 기준 안에서만 구성한다.

---

## 7.3 데이터 소스 분리 원칙
본 시스템은 여러 source_type을 다루지만 동일하게 처리하지 않는다.

### PDF / HTML
- 설명형, 안내형, FAQ형 지식 소스
- Parse 및 RAG indexing 대상

### CSV
- 구조화된 매핑/마스터 데이터
- Parse IR 대상이 아님
- structured no-parse → canonicalization → entity/relation/index row 생성 대상

### API
- 시험일정 / 접수일정 / 지원 링크 등 최신성 정보
- 현재는 후속 스프린트 범위

## 8. 핵심 기능 범위

## 8.1 현재 범위(MVP 범위)

### 기능 1. 위험군 기반 추천 진입
사용자로부터 위험군 관련 입력 또는 사전 판별 결과를 받아  
추천에 사용할 기준 단계(risk_stage)를 결정한다.

### 기능 2. 자격증 후보 추천
사용자의 위험군 단계, 관심 도메인, 관심 직무를 바탕으로  
관련 자격증 후보를 추천한다.

추천 결과는 최소한 아래 정보를 포함해야 한다.

- 자격증명
- 관련 직무
- 관련 도메인
- 추천 근거 요약
- 연결 가능한 로드맵 단계

### 기능 3. 로드맵 제안
추천된 자격증 또는 도메인을 기반으로  
위험군 단계에 맞는 로드맵 단계를 제안한다.

### 기능 4. 설명 근거 검색
PDF / HTML 기반 지식 소스에서 FAQ, 안내문, 설명 텍스트를 검색하여  
추천 결과에 설명 근거를 붙일 수 있도록 한다.

### 기능 5. 관리자/운영용 데이터 정비 기반
CSV canonicalization을 통해
- 자격증
- 직무
- 도메인
- 위험군
- 로드맵 단계
간의 관계를 정규화한다.

---

## 8.2 후속 범위(Post-MVP)

### 기능 6. 시험 일정 연동
자격증별 시험 일정 연결

### 기능 7. 접수 일정 연동
자격증별 접수 시작/종료 일정 연결

### 기능 8. 지원 링크 연결
기관별 공식 링크, 접수 링크, 안내 페이지 연결

### 기능 9. 고도화된 retrieval / reranker
- hybrid tuning
- reranker 학습
- 고도화된 query rewrite
- latency/cost 최적화

---

## 9. 비범위(Out of Scope)

현재 PRD 기준에서 아래 항목은 제품 범위 밖으로 둔다.

- 위험군 2~4단계의 최종 정책 정의
- 시험일정 API 실연동
- 접수일정 API 실연동
- 지원 링크 최신화 자동화
- 하이브리드 가중치 자동 튜닝
- reranker 학습 및 배포
- 완전한 상담형 대화 에이전트
- 사용자 계정 기반 개인화 히스토리 고도화

## 10. 사용자 시나리오

## 시나리오 1. 방향 탐색형 사용자
1. 사용자는 자신의 현재 상태를 입력한다.
2. 시스템은 위험군 단계 기준을 바탕으로 추천 흐름을 시작한다.
3. 시스템은 관련 자격증 후보와 관련 직무/도메인을 함께 제시한다.
4. 사용자는 왜 이 자격증이 추천되었는지 설명을 확인한다.
5. 사용자는 로드맵 단계를 확인하고 다음 행동을 정한다.

## 시나리오 2. 직무 지향형 사용자
1. 사용자는 관심 직무를 선택하거나 입력한다.
2. 시스템은 허용된 직무 taxonomy에 매핑한다.
3. 관련 도메인과 자격증 후보를 추천한다.
4. 위험군 단계에 맞춰 로드맵 단계를 다르게 제시한다.

## 시나리오 3. 자격증 탐색형 사용자
1. 사용자는 특정 자격증 또는 자격증군을 탐색한다.
2. 시스템은 해당 자격증의 관련 직무/도메인/추천 맥락을 보여준다.
3. FAQ 또는 안내문에서 관련 설명 근거를 제공한다.
4. 추후 일정 정보가 붙으면 시험/접수 일정까지 확인할 수 있다.

## 시나리오 4. 운영자/관리자
1. 운영자는 CSV 마스터/매핑 데이터를 업데이트한다.
2. 시스템은 canonicalization을 수행한다.
3. entity / relation / candidate row를 재생성한다.
4. 추천용 인덱스와 설명용 인덱스를 갱신한다.

## 11. 사용자 스토리

### 사용자 스토리 1
청년 사용자로서,  
내 상황에 맞는 자격증을 추천받아  
막연한 탐색이 아니라 실제 준비 가능한 선택지를 보고 싶다.

### 사용자 스토리 2
직무를 고민하는 사용자로서,  
관심 직무와 연결된 자격증과 도메인을 함께 보고  
왜 이 자격증이 필요한지 이해하고 싶다.

### 사용자 스토리 3
위험도가 높은 사용자로서,  
단순 추천이 아니라 단계별 로드맵을 받아  
무엇부터 해야 할지 순서를 알고 싶다.

### 사용자 스토리 4
운영자로서,  
CSV 기반 마스터/매핑 데이터를 정규화하여  
추천 결과의 일관성과 품질을 유지하고 싶다.

## 12. 제품 성공 기준

## 12.1 정성 기준
- 사용자가 추천 결과를 “이해 가능하다”고 느껴야 한다.
- 추천 이유와 연결 관계가 납득 가능해야 한다.
- 로드맵이 추상적 조언이 아니라 다음 행동으로 읽혀야 한다.

## 12.2 정량 기준(초기)
초기 단계에서는 아래 지표를 우선 관리한다.

- 추천 결과 생성 성공률
- canonicalization validation 통과율
- taxonomy 밖 라벨 생성 0건
- 설명 근거 검색 성공률
- 사용 시나리오별 추천 결과 completeness

추후 Retrieval / Evaluation 문서에서 세부 metric을 확정한다.

---

## 13. 제품 제약 조건

### 13.1 taxonomy 제약
- `related_domains`는 허용된 도메인 세부 라벨만 사용
- `related_jobs`는 허용된 직무 세부 라벨만 사용

### 13.2 데이터 처리 제약
- CSV는 Parse IR을 만들지 않는다.
- CSV는 canonicalization 전용 경로를 따른다.
- PDF / HTML과 CSV / API는 동일하게 처리하지 않는다.

### 13.3 일정 정보 제약
- 시험일정 / 접수일정 / 지원 링크는 후속 스프린트 범위다.
- 현재는 관련 필드를 reserved 또는 null 상태로 유지할 수 있다.

### 13.4 문서 우선 제약
- 기능 추가 전 루트 md 문서를 먼저 수정한다.
- 구현은 문서 기준선을 따라야 한다.

---

## 14. MVP 정의

현재 MVP는 아래 조건을 만족하면 된다.

1. 위험군 단계 정보를 기반으로 추천 흐름이 시작될 수 있다.
2. 자격증-직무-도메인 관계가 canonicalized 데이터로 정리된다.
3. `certificate_candidate_row` 수준의 추천 후보 구조를 만들 수 있다.
4. PDF / HTML 지식 소스를 Parse 및 indexing 할 수 있다.
5. 추천 결과와 설명 근거를 연결할 수 있다.
6. 일정 API 없이도 추천/로드맵 중심 흐름이 동작한다.

## 17. 최종 요약

이 프로젝트는 청년의 위험군 단계와 관심 직무/도메인을 기반으로  
자격증 추천과 로드맵 제안을 연결하는 시스템이다.

현재 제품의 핵심은 아래 두 축이다.

1. **CSV canonicalization 기반 추천 구조 정비**
2. **PDF / HTML 기반 설명 근거 검색 구조 정비**

즉, 지금 단계의 제품 목표는  
“일정까지 포함한 완성형 서비스”가 아니라  
**위험군 맞춤 추천 + 로드맵 제안이 가능한 구조를 먼저 안정적으로 만드는 것**이다.
