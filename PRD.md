- 현재 해결해야 하는 문제

이 문서는 다음을 직접 정의하지 않는다.

## 2. 제품 한 줄 정의

청년의 위험군 단계와 관심 직무/도메인을 바탕으로,  
관련 자격증을 추천하고 단계에 맞는 로드맵을 제안하는 시스템을 구축한다.  
추후 시험 일정, 접수 일정, 지원 링크까지 연결할 수 있는 확장 구조를 목표로 한다.

## 3. Active Problems

현재 제품이 우선 해결해야 하는 문제는 아래와 같다.

### 3.1 개인 상태가 반영되지 않는 추천
기존 자격증 탐색은 사용자 상태를 충분히 반영하지 못한다.

- 사용자의 현재 위험 수준이 반영되지 않는다.
- 단순 인기 자격증 또는 일반 추천 수준에 머무른다.
- “내 상황에서 지금 무엇이 적절한가”에 답하기 어렵다.

### 3.2 자격증-직무-도메인-로드맵이 분리되어 있음
사용자는 자격증을 알아도 그것이 어떤 직무와 연결되고, 어떤 도메인으로 이어지며, 어떤 순서로 준비해야 하는지 이해하기 어렵다.

- 자격증 정보는 있어도 관련 직무가 명확하지 않다.
- 어떤 도메인으로 이어지는지 구조적으로 보이지 않는다.
- 어떤 단계에서 무엇을 먼저 준비해야 하는지 알기 어렵다.

### 3.3 설명형 문서와 구조화 데이터가 분리되어 있음
설명형 문서와 구조화 데이터가 별개의 소스로 존재하여 하나의 사용자 흐름으로 연결되지 않는다.

- PDF 안내문
- HTML 설명 페이지
- CSV 마스터/매핑 데이터
- 추후 일정/링크 데이터

이들이 분리되어 있어 사용자는 “추천”과 “설명”을 하나의 흐름으로 받기 어렵다.

### 3.4 추천 결과가 다음 행동으로 이어지지 않음
기존 탐색은 결과를 보여주는 수준에서 끝나기 쉽다.

- 왜 이 자격증이 적합한지 이해하기 어렵다.
- 추천 결과 이후 무엇을 해야 하는지 모호하다.
- 단계형 로드맵이 없으면 행동으로 이어지기 어렵다.

## 4. 타깃 사용자

### 4.1 1차 타깃 사용자
취업 또는 진로 준비 중인 청년 사용자

대표 예시:
- 무엇부터 준비해야 할지 모르는 사용자
- 관심 직무는 있지만 연결 자격증을 모르는 사용자
- 취업 의지는 있으나 준비 수준이나 환경 제약 때문에 어려움을 겪는 사용자

### 4.2 2차 타깃 사용자
구조화 데이터와 추천 기준을 운영하는 관리자/기획자

대표 예시:
- 자격증-직무-도메인 매핑 데이터를 정리하는 관리자
- 위험군 단계별 로드맵 기준을 운영하는 기획자
- 추천 규칙과 데이터 품질을 검토하는 협업자

## 5. 핵심 사용자 가치

### 5.1 내 상태에 맞는 추천
사용자는 “모두에게 좋은 자격증”이 아니라 **현재 내 단계에서 의미 있는 자격증**을 추천받아야 한다.

### 5.2 이해 가능한 연결
사용자는 추천 결과를 볼 때 아래를 이해할 수 있어야 한다.

- 어떤 직무와 연결되는지
- 어떤 도메인과 관련되는지
- 왜 지금 추천되는지

### 5.3 다음 행동의 명확성
사용자는 추천 결과를 본 뒤 아래를 구분할 수 있어야 한다.

- 지금 준비할 것
- 다음 단계에서 준비할 것
- 추후 확인할 것

## 6. 제품 목표

본 제품의 현재 목표는 아래 3가지를 하나의 흐름으로 연결하는 것이다.

### 6.1 위험군 기반 개인화
사용자를 1단계~5단계 위험군 체계로 관리하고, 단계에 따라 추천 우선순위와 로드맵 구조를 다르게 제공한다.

### 6.2 구조적 추천 제공
자격증을 단일 객체가 아니라 아래 관계 안에서 추천한다.

- 자격증 ↔ 직무
- 자격증 ↔ 도메인
- 위험군 ↔ 로드맵 단계

### 6.3 설명 가능한 결과 제공
단순 추천 목록이 아니라 아래를 함께 제공한다.

- 추천 이유
- 관련 직무/도메인
- 연결 가능한 로드맵 단계
- 문서 기반 설명 근거

---

## 7. 핵심 제품 개념

### 7.1 위험군 단계 체계
위험군은 **1단계 ~ 5단계** 구조로 관리한다.

- **1단계**: 취업 안정권
- **5단계**: 취업을 하고 싶어도 하기 어려운 가장 높은 위험군
- **2~4단계**: 세부 의미는 후속 정책 문서에서 확정

현재 PRD에서는 위험군을 **단계형 개인화 기준**으로만 고정한다.

### 7.2 taxonomy 고정
추천 결과의 정합성을 위해 자유 텍스트 라벨을 허용하지 않는다.

- `related_domains`는 domain taxonomy의 세부 라벨만 사용
- `related_jobs`는 희망 직무 taxonomy의 세부 직무만 사용

### 7.3 source_type 분리
source_type에 따라 제품 내부 처리 기준을 구분한다.

- PDF / HTML: 설명 근거 검색용 지식 소스
- CSV: 구조적 추천 기반 데이터 소스
- API: 추후 일정/링크 최신성 소스

## 8. MVP 범위

현재 MVP는 아래 범위까지만 포함한다.

### 8.1 위험군 기반 추천 진입
사용자의 위험군 단계 또는 관련 입력을 받아 추천 흐름을 시작한다.

### 8.2 자격증 후보 추천
위험군 단계, 관심 직무, 관심 도메인을 바탕으로 관련 자격증 후보를 추천한다.

최소 포함 정보:
- 자격증명
- 관련 직무
- 관련 도메인
- 추천 이유 요약
- 연결 가능한 로드맵 단계

### 8.3 로드맵 제안
추천된 자격증 또는 도메인을 기반으로 위험군 단계에 맞는 로드맵 단계를 제안한다.

### 8.4 설명 근거 검색
PDF / HTML 기반 공식 문서에서 설명 근거를 검색해 추천 결과에 붙일 수 있게 한다.

### 8.5 구조화 데이터 정비
CSV canonicalization을 통해 자격증, 직무, 도메인, 위험군, 로드맵 단계 간 관계를 정규화한다.

## 11. 대표 사용자 시나리오

### 시나리오 1. 방향 탐색형 사용자
1. 사용자가 현재 상태를 입력한다.
2. 시스템이 위험군 기준으로 추천 흐름을 시작한다.
3. 시스템이 자격증 후보와 관련 직무/도메인을 함께 제시한다.
4. 사용자가 추천 이유와 로드맵 단계를 확인한다.

### 시나리오 2. 직무 지향형 사용자
1. 사용자가 관심 직무를 선택한다.
2. 시스템이 허용된 직무 taxonomy에 매핑한다.
3. 관련 도메인과 자격증 후보를 추천한다.
4. 위험군 단계에 맞는 로드맵을 제시한다.

### 시나리오 3. 자격증 탐색형 사용자
1. 사용자가 특정 자격증 또는 자격증군을 탐색한다.
2. 시스템이 관련 직무/도메인/추천 맥락을 보여준다.
3. 문서 기반 설명 근거를 제공한다.
4. 후속 단계에서는 일정 정보까지 연결한다.

### 시나리오 4. 운영자
1. 운영자가 CSV 마스터/매핑 데이터를 갱신한다.
2. 시스템이 canonicalization을 수행한다.
3. entity / relation / candidate row를 재생성한다.
4. 추천과 설명 검색의 기반 데이터를 갱신한다.

## 12. 사용자 스토리

### 사용자 스토리 1
청년 사용자로서,  
내 상황에 맞는 자격증을 추천받아 실제 준비 가능한 선택지를 보고 싶다.

직무를 고민하는 사용자로서,

위험도가 높은 사용자로서,  
단순 추천이 아니라 단계별 로드맵을 받아 무엇부터 해야 할지 알고 싶다.

### 사용자 스토리 4
운영자로서,  
CSV 기반 마스터/매핑 데이터를 정규화해 추천 결과의 일관성과 품질을 유지하고 싶다.

## 13. 성공 기준

### 13.1 정성 기준
- 사용자가 추천 결과를 이해 가능하다고 느껴야 한다.
- 추천 이유와 연결 관계가 납득 가능해야 한다.
- 로드맵이 다음 행동으로 읽혀야 한다.

### 13.2 정량 기준(초기)
초기 단계에서는 아래를 우선 관리한다.

- 추천 결과 생성 성공률
- canonicalization validation 통과율
- taxonomy 밖 라벨 생성 0건
- 설명 근거 검색 성공률
- 시나리오별 결과 completeness

세부 metric 정의는 `EVALUATION_GUIDELINE.md`에서 확정한다.

## 14. 제품 제약

### 14.1 taxonomy 제약
- `related_domains`는 허용된 도메인 세부 라벨만 사용
- `related_jobs`는 허용된 직무 세부 라벨만 사용

### 14.2 데이터 처리 제약
- CSV는 Parse IR을 만들지 않는다.
- CSV는 canonicalization 전용 경로를 따른다.
- PDF / HTML과 CSV / API는 동일하게 처리하지 않는다.

### 14.3 일정 정보 제약
- 일정 / 링크는 후속 스프린트 범위다.
- 현재는 관련 필드를 reserved 또는 null 상태로 둘 수 있다.

### 14.4 문서 우선 제약
- 기능 추가 전 루트 문서를 먼저 수정한다.
- 구현은 문서 기준선을 따라야 한다.

## 15. MVP 완료 조건

현재 MVP는 아래 조건을 만족하면 된다.

1. 위험군 단계 정보를 기반으로 추천 흐름이 시작될 수 있다.
2. 자격증-직무-도메인 관계가 canonicalized 데이터로 정리된다.
3. `certificate_candidate_row` 수준의 추천 후보 구조를 만들 수 있다.
4. PDF / HTML 문서를 Parse 및 indexing 할 수 있다.
5. 추천 결과와 설명 근거를 연결할 수 있다.
6. 일정 API 없이도 추천/로드맵 중심 흐름이 동작한다.

## 18. 최종 요약

이 프로젝트는 청년의 위험군 단계와 관심 직무/도메인을 기반으로  
자격증 추천과 로드맵 제안을 연결하는 시스템이다.

현재 제품의 핵심은 아래 두 축이다.

1. **CSV canonicalization 기반 추천 구조 정비**
2. **PDF / HTML 기반 설명 근거 검색 구조 정비**
